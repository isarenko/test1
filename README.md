СПП Service (Backend)
Описание проекта
Прототип backend-сервиса для управления иерархическим справочником СПП и выполнения каскадного распределения суммы по древовидной структуре.

Функциональность:
получение дерева СПП с фильтрацией по датам (версионность)
- распределение суммы по выбранным узлам
- рекурсивный расчет значений по дереву
- сохранение результатов расчётов
- хранение временных результатов в Redis
- перенос результатов в PostgreSQL
- экспорт результатов в Excel
- SSE-уведомления о событиях

Архитектура
Сервис построен по принципу разделения:
PostgreSQL — источник данных (source of truth)
Redis — временное хранилище расчетов
API — слой вычислений и агрегации
SSE — простой real-time канал уведомлений

Поток данных
1. /calculate
загрузка дерева из PostgreSQL
выполнение алгоритма распределения
сохранение результата в Redis (TTL 1 час)
2. /save/{calc_id}
чтение из Redis
запись результата в PostgreSQL (JSONB)
3. /stream/{session_id}
подписка на события сохранения (SSE)

Модель данных
spp_elements
Иерархический справочник:
parent_id — связь дерево (adjacency list)
valid_from / valid_to — версионность
is_active — активность записи

calculations
Результаты расчетов:
calc_id (UUID)
session_id
status
data (JSONB snapshot)
created_at

departments / department_spp_links
Справочники отделов и связей со СПП (заложено в ТЗ, в расчётах не используется)

Алгоритм распределения
Используется DFS (обход в глубину):
выбранные узлы делят сумму поровну
каждый узел рекурсивно распределяет свою долю между детьми
листья получают финальные значения
родитель агрегирует сумму детей
Округление: 2 знака после запятой.

Redis → PostgreSQL → Real-time
1. Redis
Используется как cache слоя вычислений:
хранение результата расчёта
TTL 3600 секунд
быстрый доступ до сохранения

2. PostgreSQL
Основное хранилище:
справочник СПП
сохранённые расчёты
JSONB snapshot результата

3. Real-time (SSE)
подписка на канал calc_updates
уведомление клиентов о сохранении расчёта
упрощённая альтернатива WebSockets

Принятые архитектурные решения
- adjacency list для дерева СПП
- Redis как временный слой между расчетом и БД
- синхронный расчет без очередей (упрощение MVP)
- SSE вместо WebSocket для снижения сложности
- хранение результатов расчета в JSONB (гибкость схемы)
- Docker Compose как единая точка запуска
